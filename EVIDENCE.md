## Publication rules

Before moving evidence from private to public, sanitize:

- source file names if they reveal implementation internals
- local paths
- credentials, tokens, or environment details
- exact private prompts
- private backlog items unrelated to the proof claim
- detailed model-routing heuristics that should remain proprietary
- implementation code or architecture details that make the private engine easy to clone

## Priority evidence gaps

1. Publish a sanitized run-proof JSON example.
2. Add redacted screenshots of PR summaries or validation output.
3. Add a clear public architecture diagram that exposes the control flow but not implementation.
4. Add a recurring proof update process after major private milestones.
5. Publish a third sanitized case study for token-efficient context assembly or structured evidence packets.
